# Productos de Base Tecnológica - Primera Clase

## Introducción: ¿Qué es un Producto de Base Tecnológica?

### Definición Informal
    -> Ejemplo visto: Uber
    -> Es un producto basado en tecnología
    -> Se llama así a la materia que está basado en la tecnología
    -> Si quitas esa tecnología en general no es posible
    -> Se habla del producto más genérico
    -> No solamente la parte del software o la tecnología
    -> Sino de todo lo que hay alrededor del producto
    -> Los servicios, el negocio

### Ejemplo Uber
    -> Aplicación del conductor
    -> Aplicación del pasajero
    -> Equipo de soporte atrás
    -> Equipo administrativo
    -> Gente de marketing
    -> Todo eso forma parte del producto

### Definición Formal (Libro de Spencer)
    -> Un producto de base tecnológica se habla de un producto holístico (integral)
    -> Contiene estos puntos:
        -> Toda la funcionalidad que ofrece el producto
        -> Están impulsados por la tecnología
        -> Eso hace posible que exista
        -> Tiene muy buen diseño y una muy buena experiencia
        -> Cuando hablamos de experiencia usuaria podemos ver dos cosas:
            -> La experiencia de la tecnología asociada (si tiene una app asociada)
            -> Pero casi habla también de la experiencia completa total
        -> Ejemplo Pedidos Ya:
            -> Es importante la experiencia usuario del app
            -> Pero si tienen algún problema con el pedido que no llega
            -> Es importante la experiencia con el soporte
            -> La experiencia que le devuelvan rápidamente
            -> Que le den un cupón por si tuvo algún primer pedido
            -> Todo eso forma parte de la experiencia
        -> Tiene un modelo de negocio atrás
            -> Lo que le da viabilidad de negocio
    -> El valor en realidad no está en la tecnología en sí
    -> Nosotros como ingenieros a veces nos encontramos mucho en la tecnología y en el desarrollo del app
    -> Sino que es el valor que damos con eso a las personas y empresas

## Dos Fases del Desarrollo de Software

### Discovery (Descubrimiento)
    -> Objetivo: definir qué es lo que hay que construir
    -> Analizar el problema con usuarios y validar la solución
    -> Definir qué funcionalidades vamos a construir
    -> Construir lo que es definir lo que hay que hacer de valor usuario que sea deseable
    -> Esta es la fase en la que nos enfocamos en esta materia

### Delivery (Entrega)
    -> Lo que han estudiado más en esta carrera
    -> Proceso de gestión del desarrollo
    -> Ejemplo ISA 1: arrancaron con necesidad ya definida por alguien
        -> Product owner que arranca de un requerimiento
        -> Lo carga en forma de user story
        -> Después todo el desarrollo con SCRUM
        -> Story pointing
        -> Ceremonias: sprint planning, daily standup, retrospectiva
    -> Ejemplo ISA 2: vieron esto implementado con tecnologías
        -> Integración continua
        -> Entrega continua
        -> Automatización del pipeline

### Cuando Trabajamos en Proyectos vs Productos

#### En Proyectos
    -> Nos preguntan al usuario cómo quiere modificar su carrito de compras
    -> Cambian el requerimiento
    -> Pero arrancan por un requerimiento específico

#### En Productos
    -> No parten de un usuario que les dice lo que quiere
    -> Parten de un problema
    -> Problema que alguien quiere resolver o que vale la pena resolver
    -> O que ustedes deciden que vale la pena resolver
    -> Los emprendimientos surgen detectando un problema
    -> No van directo a desarrollar
    -> Tienen que hacer el proceso de Discovery primero
    -> Hay que hablar con los usuarios primero
    -> Identificar todos los usuarios involucrados
    -> Ejemplo Uber:
        -> Hablar con gente que tiene temas de locomoción
        -> Hablar con gente que quiere manejar un taxi
        -> Hablar con la gente de la intendencia, organizaciones públicas
        -> Saber la viabilidad
    -> Vamos a prototipar porque normalmente lo primero que construimos no es lo que realmente soluciona el problema
    -> El foco de la materia es el Discovery
    -> Después cuando ya sabemos lo que tenemos que construir pasamos al proceso de Delivery

## Las Cuatro Dimensiones de un Producto

### Primera Dimensión: Deseable
    -> Que alguien lo quiera
    -> Que resuelva un problema que es pertinente a resolver
    -> Que además esté dispuesto a pagar por esa solución

### Segunda Dimensión: Viable
    -> Que alguien quiera pagar por esa solución
    -> Que eso que va a pagar sea viable para poder mantener este negocio atrás del producto de base tecnológica

### Tercera Dimensión: Usable
    -> La experiencia usuario es fundamental para cualquier producto de software o producto de base tecnológica

### Cuarta Dimensión: Feasible (Factible)
    -> Pueden tener muchas ideas espectaculares con tecnología bárbara
    -> Pero a veces no son viables porque la tecnología no está madura
    -> La tecnología que quieren usar no existe
    -> Si le existe, no se les va a ocurrir
    -> Pero este acaba que no la conozcan

## El Equipo de Producto

### Roles Principales

#### Product Manager
    -> Persona que tiene tres patas
    -> Tiene una pata conectando al mercado
    -> Con los usuarios: entender el problema y las necesidades
    -> Con el negocio: lo que necesitan los usuarios puede haber un negocio atrás
    -> Trabaja con el equipo de construcción del software
    -> El product manager decide qué vale la pena construir
    -> Porque él solo tiene mucha necesidad
    -> Define junto al equipo qué vale la pena construir
    -> Ejemplo: un problema empiezan a desmenuzar, empiezan a hacer discovery
    -> Dicen podría solucionar esto, esto, esto, esto, esto
    -> Pero no todo se puede solucionar
    -> Entonces el product manager define qué vale la pena construir

#### Product Designer
    -> La experiencia usuario es fundamental
    -> Alguien que nos apoya a diseñar la solución y la experiencia usuario
    -> La experiencia no solo del producto tecnológico
    -> Sino también de todo lo que hay alrededor

#### Ingenieros/Desarrolladores
    -> Factibilidad técnica
    -> Equipo de ingenieros desarrolladores
    -> Nosotros estamos más que nada acá
    -> Aunque nuestra carrera puede evolucionar a otros roles

### Trabajo del Equipo
    -> Trabajan juntos
    -> Cuando hay que definir algo, se juntan y trabajan bastante juntos
    -> Antes no era tan así
    -> Antes teníamos la tecnología por un lado
    -> Tenías a product manager pero ni siquiera capaz se había probado un manager con ese nombre
    -> Product Designer era un diseñador gráfico
    -> Nosotros desarrolladores hacíamos algo y le pedíamos que haga la parte gráfica
    -> Ahora estos tres empezaron a trabajar juntos

### Otros Roles (Empresas Más Grandes)
    -> User researchers: gente especializada en el discovery y con los usuarios
    -> Analistas de datos: cuando se hace research para entender las necesidades o evaluar prototipos
        -> Se pueden recolectar muchos datos
        -> Esta gente analiza esos datos para generar insights
        -> Medir el impacto
        -> Ver qué ayuda al product manager a tomar decisiones del producto
    -> Muchos de los datos que se generan a veces se generan automáticamente
    -> Ejemplo Netflix: cuántos usuarios son, que ven que no ven, cuánto tiempo pasan
        -> Empieza una película son todos métricas que se recogen automáticamente
    -> Equipo de marketing que están como gerencia a eso

### Valoración del Product Manager con Base Tecnológica
    -> En los productos de base tecnológica se valora mucho que el product manager tenga una base tecnológica
    -> A veces los product managers en otros negocios incluso que tienen base tecnológica del software no vienen del desarrollo
    -> Y no está mal
    -> Pero las empresas que tienen una base de tecnología fuerte se valora mucho eso
    -> Hay bastante mercado para gente de base tecnológica para ahí
    -> Tiene que gustar obviamente

### Caminos hacia Product Manager
    -> Dos formas que se conocen
    -> Una: empezar como desarrollador
        -> Ejemplo del profesor: empezó como desarrollador perdón un producto
        -> Ese producto se fue cambiando
        -> Pasó a hacer product manager igual
        -> En Uruguay en esa época era product manager y seguía desarrollando dos días
        -> Después hubo una spin-off
        -> Ahí hizo product manager de un nuevo producto
        -> Después otra chica entró al trabajo conmigo
        -> Quedó más como líder de equipo de desarrollo e investigación
    -> Otra: puede haber llamado para product owner
        -> Capacidad que más enfocado al cliente
        -> Ejemplo: en ISA 1 vieron el producto owner
        -> Product owner está dedicado a un módulo o a un producto muy específico
        -> Trabaja con un equipo de desarrollo
        -> Tiene contacto tanto con los clientes o los usuarios y con el equipo de desarrollo
        -> En una empresa puede haber un solo producto owner
        -> Entonces tenemos product manager acá
        -> El product manager está más orientado al negocio
        -> Está más preocupado orientado al negocio al mercado
        -> También contacta con los usuarios porque acá cuando empieza el product owner ya tiene un conjunto de clientes usuarios
        -> Y ya el product owner se le da se atiende a esos clientes y usuarios
        -> Pero el product manager mira más el negocio, el mercado para ampliar el conjunto de clientes
        -> Trabaja también con los product owners
        -> Si hay product owners el product manager a veces no hay los dos es el mismo persona
        -> A veces pasa una empresa que tiene más de un producto
        -> Entonces este product manager también está trabajando con otro producto que tiene todo esto mismo
        -> Capacidad que son productos relacionados pero no son el mismo
    -> También hay algunas empresas que piden producto owner junior
        -> Capacidad que tenés un poco de experiencia no más en desarrollo
        -> Puedes entrar como producto owner junior
        -> Las empresas tienen un product owner que lo quieren pasar acá porque no tienen esto
        -> Empiezan a trabajar como product owner junior
        -> Trabajan un tiempo así
        -> Después te lo sacan de acá y lo suben

## Misioneros vs Mercenarios

### Misioneros
    -> Están motivados por el propósito y la visión del producto
    -> Quieren resolver problemas reales y generar impacto
    -> Ven el trabajo como una contribución a algo más grande
    -> En general son autónomos y comprometidos

### Mercenarios
    -> Están ahí solo para cumplir lo que se les pide
    -> No se involucran en el para qué
    -> No está mal
    -> No se conectan con la visión del producto ni con el valor para el usuario
    -> Se limitan a ejecutar sin entender mucho el impacto
    -> Cuando trabajan en la parte que trabajan siempre entender para qué estoy haciendo esto es bueno
    -> Pero a veces en estos equipos en esta parte sobre todo en delivery ya tenemos el descubrimiento hecho
    -> En particular cuando empiezan a trabajar o los primeros años tienen un producto owner que ya hizo el análisis que ustedes van a hacer con la tecnología
    -> Tratar de resolver lo que les dice el product owner lo mejor posible
    -> Igual está bueno saber para qué
    -> Porque si pueden sugerir mejoras esperamos un buen product owner el decir que tienen
    -> A veces va directamente acá debería bien el para qué lo desarrollado

## Características del Equipo de Producto

### Multifuncional
    -> Ya dijimos que mezcla negocios, diseño y tecnología

### Duradero
    -> La idea es que el equipo de producto se mantengan el tiempo del producto
    -> Perdón a diferencia cuando hace proyecto que vos empezar un proyecto
    -> Si hace un equipo empezó otro proyecto terminar empezó todo el proyecto
    -> Y capaz que sin su tal la idea es que a menos que ustedes si están trabajando en un equipo de producto y se vayan en otra empresa
    -> La idea es mantenerlos en el mismo equipo porque es un juego infinito
    -> Se acuerdan lo de juego finitos vs infinitos
    -> Cuál es infinito: esto está bueno
    -> Cuanto más para el du hacer el equipo mejor

### Equipos de Productos vs Equipos de Funcionalidades

#### Equipos de Productos
    -> Sirven problemas que deben resolver
    -> Tienen autonomía para descubrir el problema y la solución
    -> Trabajan más en discovery

#### Equipos de Funcionalidades
    -> Son los que trabajan más en delivery
    -> Tapan de acá para acá
    -> Más bien reciben requerimientos
    -> Generalmente es en forma de roadmap
    -> Se van por roadmap para hacer eso

## El Discovery: Building the Right Thing

### Qué es el Discovery
    -> Proceso que se resume en una frase en inglés: building the right thing
    -> En español: lo correcto, lo que sirve
    -> En el discovery se asegura que estemos construyendo lo correcto

### La Frase de Henry Ford
    -> Ejemplo de automóviles cuando la gente se movilizaba en caballos
    -> Frase: "Si hubiera preguntado a mis clientes qué querían, me habrían pedido caballos más rápidos"
    -> Problema: la gente le pedía caballos más rápidos
    -> No se les ocurría que podría haber algo nuevo
    -> Ellos partían de lo que conocían
    -> Conocían solo caballos y querían más rápido
    -> No se entiende para que no nos haya movido mi cliente
    -> Traduciendo el problema de ese pedido está perfecto
    -> Lo que dijeron dicho en otras palabras los usuarios le estaban pidiendo una solución
    -> No le estaban diciendo su problema
    -> Estaban diciendo una solución

### Qué Hacer Cuando los Usuarios Piden Soluciones
    -> Si tenían que tener en cuenta que los clientes muchas veces no saben lo que es posible
    -> Dicen caballos más rápidos porque es lo que se les ocurre
    -> Sin embargo nosotros que participamos en discovery y conocemos la tecnología
    -> Nos capacita para encontrar soluciones mejores a los que se le puede ocurrir cliente
    -> El cliente puede tener su expertise en otro lado en otra área pero no en esta
    -> Tenemos nosotros en ustedes
    -> El usuario siempre nos va a dar soluciones
    -> El 99% de las veces no va a decir yo quiero tal cosa
    -> Cuando usted se tal cosa dicen pa me está pidiendo una solución
    -> Lo que nosotros tenemos que hacer es a partir de lo que nos pide el usuario
    -> No lo tenemos que decir no me des una solución
    -> Lo que tenemos que pedirle a los usuarios es lo que tenemos que lograr
    -> Tener la capacidad para que a partir de lo que me pidió el usuario descubrir cuál es el problema
    -> País que el caso de los caballos más rápido será fácil
    -> Pero vamos a ver que hay otros casos que no es tan fácil

### Nuestro Error y Nuestro Valor
    -> Error no es preguntar qué quieren
    -> No es preguntar sus requerimientos
    -> Hay forma de preguntar sí
    -> Pero porque en general esa pregunta respondía con soluciones
    -> Nuestro valor está justamente en descubrir la necesidad profunda
    -> Hay una imagen de un iceberg
    -> Lo que usuario les va a pedir es esto (arriba del agua)
    -> Que ustedes tienen que llegar acá abajo (debajo del agua)
    -> Entender cosas que hay acá que son la necesidad más profunda o necesidades
    -> Las necesidades el contexto en que tienen ese problema
    -> Quiénes son las que tienen ese problema por qué lo tienen
    -> Si tiene una solución por qué
    -> Por qué van a decir la solución es otro que vamos a dar

## El Proceso de Design Thinking: Doble Diamante

### Estructura General
    -> Primero: análisis del problema (divergente)
    -> Después: definición del problema (convergente)
    -> Después: ideación de soluciones (divergente)
    -> Después: definición de la solución (convergente)

### Divergente vs Convergente
    -> Divergente: primero yo analizo observo trato de recoger datos todos
    -> Después empiezo a hacer a quedarme con lo que realmente me parece que sirve que representa bien el problema
    -> Convergente: después empiezo a hacer a a quedarme con lo que realmente me parece que sirve

### Primer Diamante del Problema

#### Empatizar
    -> Técnica para empatizar
    -> No es decir ta empate cenaban esto y listo
    -> Hay técnicas que los ayudan a hacer esto de mejor manera
    -> En empatizar ella hizo todo esto (investigación de campo)

#### Definir
    -> Acabamos a definir el problema a resolver
    -> No definir la solución seguir por ver si se confunde
    -> Estructura de POV (Point of View) para definir el problema:
        -> Las personas que tienen problema
        -> Qué necesitan
        -> El insight: por qué necesitan esto o por qué o qué es lo que yo descubrí de mi proceso de empatía
        -> Por qué necesitan eso

#### Característica de la POV
    -> No puede tener solución siempre
    -> A veces cuando hacen una POV uno tiende a decir no en este caso
    -> Pero en otros casos es fulanito de tal necesita una app para poder hacer reservas médicas sin trasladarse
    -> No eso no es una POV ese no es el problema que tiene
    -> Eso es una solución a un problema que tiene probablemente
    -> Pero si ustedes ya arrancan de la solución se pierden
    -> No pasan al idea de acá se pierden una cantidad de oportunidades de soluciones mejor

### Segundo Diamante de la Solución

#### Idear
    -> Casi vamos a empezar a pensar en la solución
    -> Vamos a idear soluciones
    -> Es un diamante divergente porque empecé a generar ideas
    -> Son varias
    -> Después es que tenés un conjunto importante de ideas donde importa más la cantidad de ideas que la calidad
    -> Se va refinando y se va definiendo qué idea vamos a prototipear y vamos a testear
    -> Hasta llegar a una solución

#### Prototipar y Testear
    -> Después que definió el problema idearon prototiparon
    -> Vean que primero llega un prototipo de baja calidad dibujos
    -> Después se hicieron algo más rudimentario
    -> Lo probaron en el campo o sea no en del escritorio o tuvieron feedback
    -> Y así hicieron varias iteraciones de esto a llegar a la solución

## Ejercicio en Grupo: Problema de la Cantina

    -> En los recreos si quieres ir a la cantina tenías que hacer tremenda fila
    -> A veces no compras cosas por ver la fila
    -> O más no te da tiempo
    -> El tiempo es crítico porque el recreo son cinco o 10 minutos
    -> O en la pausa de trabajo tenés cierto tiempo hacen 30 o 40 cinco minutos para comer
    -> Ese tiempo que perdés solo venir a hacer la fila a pedir las cosas que lleguen
    -> La ruina más y si le que si no tuviera minutos tuviera otro equipo de mundo que son seis persona pero minutos hasta nadie venda la fila
    -> Siempre que voy a comprar trabajo no ya se ya aparte dentro las son filas

    -> Estudiantes
    -> Profesores
    -> Trabajadores
